# Project Charter

## 배경

기존 `clibase`는 CLI/검증 자동화 중심 구조였고, 현재 목표는 그 위에 프로젝트/탭/모듈 기반의 워크스페이스 셸을 다시 세우는 것입니다. 다만 새 셸의 정체성은 단순 GUI가 아니라 `CLI-first automation workbench`입니다. 따라서 UI 베이스는 `ref/basic_reference`를 참고해 재구성하되, 문서 우선 개발과 단계 강제라는 운영 원칙은 유지합니다.

## 목표

- 전역 사이드바에서 프로젝트 관리와 프로젝트 전환을 담당한다.
- 글로벌 CLI가 모든 프로젝트 CLI의 기준이 된다.
- 각 프로젝트는 글로벌 CLI에서 파생된 자신의 effective CLI를 가진다.
- 각 프로젝트는 여러 탭을 가진다.
- 각 탭은 독립적인 레이아웃과 모듈 구성을 가진다.
- 모듈은 공통 카탈로그에서 선택해 프로젝트 탭에 배치한다.
- 프로젝트 관리 화면은 "실행 화면"이 아니라 프로젝트 구조를 설계하는 편집 화면이 된다.
- 최종 제품은 단순 브라우저 SPA가 아니라, 터미널/내장 브라우저/모듈 간 통신을 지원하는 데스크톱 셸을 목표로 한다.
- 모든 안정된 기능은 CLI와 AI가 같은 계약으로 제어 가능해야 한다.
- 목표 사용자는 비개발자 전체가 아니라, 약간의 코드 지식이 있는 파워유저다.

## 비목표

- 코드 한 줄도 모르는 완전한 no-code 제품을 목표로 하지 않는다.
- GUI만으로 모든 기능을 우회 조작하는 GUI-first 제품을 목표로 하지 않는다.
- 일반 소비자용 단순 앱 런처를 목표로 하지 않는다.

## 운영 원칙

- `batcli workflow start -> to-doc -> to-code -> stop` 흐름은 계속 유지한다.
- `doc/0. Governance/ssot.yaml`과 `doc/0. Governance/ddd.md`는 구현 전 반드시 먼저 갱신한다.
- 화면 구조, 용어, 엔티티가 바뀌면 코드보다 문서가 먼저 바뀌어야 한다.
- 새 UI 루트로 전환하더라도 문서 강제 시스템은 프로젝트의 핵심 규칙으로 간주한다.
- 새 기능은 GUI 노출만으로 완료되지 않으며, CLI/action 계층과 AI 제어면까지 연결되어야 한다.

## 성공 기준

- 앱이 로컬에서 바로 실행된다.
- 글로벌 CLI, 프로젝트 CLI, GUI, AI가 같은 제어 계약을 공유한다.
- 프로젝트 관리, 프로젝트 편집, 프로젝트 작업공간의 세 화면 경계가 분명하다.
- 문서만 봐도 다음 구현 순서를 판단할 수 있다.
- 터미널, 내장 브라우저, 앱 상태를 안전한 IPC로 연결하는 런타임 계획이 문서화되어 있다.
