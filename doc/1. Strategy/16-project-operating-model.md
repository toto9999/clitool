# Project Operating Model

## 목적

이 문서는 `프로젝트`가 이 제품 안에서 어떤 운영 단위인지, 그리고 글로벌 CLI, 프로젝트 CLI, GUI, AI가 어떤 관계로 동작하는지 정리한다.

## 프로젝트의 정의

이 제품에서 `Project`는 단순 저장 폴더가 아니다.

- 설정 단위
- 실행 단위
- 관찰 단위
- 자동화 단위
- 채널 노출 단위

즉 프로젝트 하나가 하나의 운영 셸이다.

## 제어 계층

### 1. Global CLI Base

- 전체 앱의 기준 제어면
- 공통 명령 구조, 공통 capability, 공통 profile을 정의
- 새 기능이 들어와도 이 계층에 투영되어야 한다
- 권위 있는 interactive host는 Textual 기반으로 구성한다

### 2. Project CLI Context

- 글로벌 CLI에서 파생된 프로젝트 전용 컨텍스트
- 프로젝트별 working directory, env, attachment, auth, channel, runtime 차이를 반영
- 사용자는 프로젝트마다 자신의 CLI 동작 방식을 세팅할 수 있다

### 3. GUI

- 프로젝트 구조 편집
- 상태 관찰
- 시각적 설정
- 로그/결과/브라우저/터미널 관찰

GUI는 실행 authority의 근원이 아니라, 같은 계약을 더 다루기 쉽게 보여주는 면이다.

### 4. AI

- 같은 CLI/action 계층을 쓰는 자동화 실행자
- 새 기능도 CLI/action projection만 되어 있으면 AI가 계속 제어 가능해야 한다
- AI 전용 숨은 제어면은 만들지 않는다

## 프로젝트를 구성하는 요소

- 프로젝트 메타데이터
- 프로젝트 CLI 파생 설정
- 탭과 레이아웃
- 모듈 인스턴스
- Skill/MCP attachment
- auth provider 선택
- ingress/channel 노출
- 로그와 분석 상태

## 프로젝트 안에서의 제어 흐름

1. 사용자는 글로벌 CLI 또는 GUI에서 프로젝트를 선택한다.
2. 시스템은 해당 프로젝트의 effective CLI context를 계산한다.
3. 사용자는 터미널, GUI, Skill, MCP, AI 중 하나로 action을 요청한다.
4. 모든 요청은 control plane을 지난다.
5. 실행 결과와 이벤트, 감사 로그는 프로젝트 기준으로 누적된다.

## 새 기능이 들어올 때의 운영 기준

새 기능은 아래 4가지를 모두 가져야 한다.

1. runtime contract
2. CLI/action projection
3. AI-usable manifest
4. GUI surface 또는 관찰 surface

이 중 하나라도 빠지면 프로젝트 운영 모델과 맞지 않는다.

## 프로젝트에서 CLI가 하는 역할

- 프로젝트 전체 도구를 한 명령 체계로 묶는다
- 브라우저, 터미널, MCP, Skill, 외부 런타임을 같은 방식으로 제어한다
- 사람이 직접 실행할 수 있다
- AI가 같은 방식으로 실행할 수 있다
- 로그, replay, audit의 기준 단위가 된다
- 멀티라인 작성, 링크/이미지 붙여넣기, reference CLI 분석 기반 재구현에 유리한 power-user surface를 제공한다

## 프로젝트에서 GUI가 하는 역할

- CLI 파생 설정을 쉽게 편집한다
- 모듈 배치와 상태를 시각화한다
- 실행 결과와 실패를 읽기 쉽게 보여준다
- 비개발자에 가까운 파워유저가 자동화를 더 쉽게 구성하게 돕는다

## 프로젝트에서 AI가 하는 역할

- 사용자가 정의한 프로젝트 구조를 이해한다
- 같은 CLI/action 계층으로 기능을 조합한다
- 새 기능이 들어와도 contract가 맞으면 별도 특례 없이 다룬다
- 로그와 상태를 읽고 다음 작업을 제안하거나 수행한다

## 이 운영 모델이 중요한 이유

- 기능이 늘어나도 제어 방식이 분산되지 않는다
- 프로젝트마다 다른 자동화 구조를 가지면서도 전체 시스템은 통일성을 유지한다
- GUI, CLI, AI가 서로 다른 시스템이 아니라 같은 시스템의 다른 면으로 동작한다
- 사용자가 커스텀한 CLI가 프로젝트 운영의 진짜 중심이 된다

## 제품 완성 기준에서의 해석

- 프로젝트를 만들 수 있는 것만으로는 부족하다
- 프로젝트를 글로벌 CLI 기준으로 파생 운영할 수 있어야 한다
- 프로젝트 안의 기능을 GUI와 AI가 같은 계약으로 제어할 수 있어야 한다
- 새 기능이 추가되어도 이 운영 모델이 깨지지 않아야 한다
