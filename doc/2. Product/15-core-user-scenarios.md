# Core User Scenarios

## 대상 사용자

- 약간의 코드 지식이 있는 파워유저
- CLI, 환경변수, 설정 파일, 간단한 스크립트를 두려워하지 않는 사용자
- 완전한 개발자는 아니지만 자동화를 직접 조립하고 싶은 사용자

## 시나리오 1. 글로벌 CLI를 기준으로 작업환경을 잡는다

사용자는 먼저 글로벌 CLI 기준을 세팅한다.

- 기본 명령 구조
- 공통 profile
- 기본 runtime
- 공통 attachment 정책

그 다음 프로젝트를 만들 때 이 글로벌 CLI를 바탕으로 프로젝트 CLI를 파생한다.

핵심 기대:
- 프로젝트마다 제로부터 다시 세팅하지 않아야 한다.
- 글로벌 기준을 바꾸면 프로젝트 운영 방식도 일관되게 관리되어야 한다.

## 시나리오 2. 프로젝트별로 다른 CLI 운영 방식을 만든다

사용자는 프로젝트마다 다른 목적을 가진다.

- 어떤 프로젝트는 문서 수집 중심
- 어떤 프로젝트는 브라우저 자동화 중심
- 어떤 프로젝트는 MCP/Skill orchestration 중심

그래서 프로젝트마다

- working directory
- env
- auth provider
- channel exposure
- module 조합
- MCP/Skill attachment

를 다르게 세팅한다.

핵심 기대:
- 프로젝트가 곧 운영 단위여야 한다.
- 프로젝트별 차이가 CLI 파생 설정으로 명확하게 관리되어야 한다.

## 시나리오 3. 터미널에서 앱 전체를 제어한다

사용자는 앱 하단 터미널이나 외부 CLI에서 명령을 실행한다.

- 브라우저 이동
- 자동화 실행
- 프로젝트 action 호출
- MCP tool 호출
- 로그 분석

핵심 기대:
- 터미널은 단순 콘솔이 아니라 앱 전체 제어의 진입점이어야 한다.
- GUI에서만 되는 기능이 있으면 안 된다.

## 시나리오 4. GUI로 구조를 쉽게 세팅한다

사용자는 CLI를 이해하지만, 모든 설정을 텍스트로만 하고 싶지는 않다.

그래서 GUI에서

- 프로젝트 구조
- 모듈 배치
- CLI 파생 설정
- auth 선택
- MCP/Skill attachment
- 채널 노출 정책

을 더 쉽게 구성한다.

핵심 기대:
- GUI는 제어 authority를 빼앗지 않고 복잡도를 낮춰야 한다.
- 시각적 구성 결과가 곧 CLI/action 시스템과 일치해야 한다.

## 시나리오 5. AI에게 같은 시스템을 맡긴다

사용자는 AI가 별도 전용 기능이 아니라 같은 시스템을 다루길 원한다.

AI는

- 프로젝트 구조를 읽고
- 가능한 action을 이해하고
- 같은 CLI/action 계층으로 실행하고
- 로그와 결과를 보고 다음 작업을 이어간다

핵심 기대:
- 새 기능이 생겨도 AI가 또 다른 전용 통로를 배워야 하면 안 된다.
- 같은 contract만 맞으면 AI가 계속 확장 가능해야 한다.

## 시나리오 6. 약간의 코드 지식으로 자동화를 확장한다

사용자는 완전한 개발자가 아니다.
하지만 다음 정도는 할 수 있다.

- CLI 옵션 수정
- preset 편집
- 간단한 YAML 수정
- env 설정
- 간단한 명령 조합

핵심 기대:
- 자동화는 전문가만 만들 수 있는 것이 아니어야 한다.
- 너무 많은 저수준 구현 지식 없이도 조립 가능해야 한다.

## 시나리오 7. 새 기능이 들어와도 운영 방식이 안 바뀐다

새 모듈, 새 MCP, 새 Skill, 새 브라우저 자동화 기능이 들어와도 사용자는 같은 방식으로 받아들여야 한다.

- CLI에 나타난다
- 프로젝트에서 붙인다
- GUI에서 보인다
- AI가 같은 방식으로 다룬다

핵심 기대:
- 기능 추가가 제어 체계의 파편화를 만들면 안 된다.
- 사용자의 학습 비용은 `새 기능 자체`에만 들고, `새 제어 방식`에는 들지 않아야 한다.
